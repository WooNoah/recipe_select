# 动态菜单分类与按数据展示 UI 设计

**背景**

当前页面只支持两个固定分类：
- `经典肉类`
- `汤品`

页面会固定渲染两张结果卡片，并在这两个分类上应用主食材配对权重。现在要支持后续逐步扩展菜单分类，包括：
- `素菜类`
- `凉拌菜类`

同时用户明确要求：
- 如果新分类在 `recipe_menu.md` 中尚未出现，则 UI 不展示该栏目
- 一旦后续在菜单中添加了新的分类 section，对应栏目应自动出现在 UI 中
- `经典肉类` 和 `汤品` 继续使用当前的主食材配对权重
- `素菜类` 需要预留自己的分类权重配置，但当前仍只抽一道，因此暂不参与联动约束

## 目标

将菜单解析、随机抽取和结果渲染改成“按 section 驱动”的结构，使页面能随着 `recipe_menu.md` 的分类变化自动增减结果卡片，而不要求每次新增分类都改前端模板。

## 方案选择

### 方案 A：固定四类，隐藏空类

将 UI 固定为四个槽位，但如果某一类没有数据则隐藏对应卡片。

优点：
- 实现简单

缺点：
- 仍然把分类写死在页面结构里
- 未来新增第五类时还要改模板和逻辑

### 方案 B：按 Markdown section 动态解析和动态渲染

把每个 `##` section 视为一个菜单分类，运行时根据实际存在的分类生成结果卡片。对已知分类应用已知规则，对未知分类使用默认独立随机。

优点：
- 真正满足“后续添加 type 后 UI 自动展示”
- 模板不再和分类数量绑定
- 规则扩展空间更大

缺点：
- 解析和渲染逻辑都要重构

### 方案 C：单独增加一个 categories 配置文件

通过额外配置定义所有分类及其渲染顺序，Markdown 只存菜名。

优点：
- 扩展性强

缺点：
- 新增分类时必须同时改菜单和配置，违背“只加数据就自动出现”的诉求

## 决策

采用方案 B。

## 数据设计

### Section 解析

`recipe_menu.md` 的每个二级标题 section 都解析为一个菜单分类，结构示意：

```js
{
  id: 'classic',
  title: '经典肉类',
  items: [
    { name: '红烧肉', metadata: { protein: 'pork' } },
  ],
}
```

说明：
- `id` 优先来自已知分类映射；未知分类回退到标题文本派生值
- `title` 用于 UI 展示
- `items` 保存菜名和元数据

### 已知分类规则

已知分类保留语义：
- `经典肉类`
- `汤品`
- `素菜类`
- `凉拌菜类`

规则：
- `经典肉类`、`汤品` 的菜项必须带 `protein=...`
- `素菜类` 的菜项如果未来启用多道素菜抽取，建议带 `veg=...`
- `凉拌菜类` 当前不要求额外元数据
- 未知分类默认不要求元数据

### 配置文件

`src/menuConfig.mjs` 保留并扩展：
- `proteinPairWeights`
- `defaultProteinPairWeight`
- `vegetablePairWeights`
- `defaultVegetablePairWeight`

说明：
- 当前只有 `proteinPairWeights` 参与实际抽样
- `vegetablePairWeights` 仅为后续“抽多道素菜”预留
- 配置文件必须包含清晰注释，说明当前哪些规则生效、哪些只是预留

## 随机抽取设计

新的抽取流程：

1. 解析 Markdown，得到一个 section 数组，保留原始顺序
2. 逐个构造各 section 的候选菜项
3. 如果存在 `经典肉类` section，则先独立抽取一道经典肉类
4. 如果存在 `汤品` section：
   - 且已抽到经典肉类时，按 `proteinPairWeights` 做加权抽样
   - 否则退回普通随机
5. 其他 section 一律独立随机抽取一道
6. 某个 section 没有候选菜时，该 section 不进入结果列表

结果是一个按 section 顺序排列的已抽中列表，而不是固定长度数组。

## UI 设计

页面结果区改为动态容器：
- 只渲染本次抽样结果中存在的 section
- 卡片标题显示该 section 的 `title`
- 卡片内容显示菜名

因此：
- 当前只有 `经典肉类` 和 `汤品` 时，页面仍显示两张卡
- 后续在 Markdown 新增 `素菜类` 或 `凉拌菜类`，对应卡片自动出现
- 后续新增未知新分类，也会以默认卡片形式出现

## 错误处理

保持这些错误：
- 缺少必需 section：`经典肉类` 和 `汤品` 至少其一必须存在，否则没有可抽内容
- `经典肉类` 或 `汤品` 项目缺失 `protein`
- 配置导致 `汤品` 在权重过滤后无可选项

不报错的情况：
- `素菜类` 或 `凉拌菜类` section 不存在
- 其他可选分类不存在
- 某个可选 section 存在但暂时没有任何条目，该 section 直接跳过

## 测试设计

逻辑测试覆盖：
- 解析任意 section，并保留标题和项目
- `素菜类`、`凉拌菜类` 缺失时不报错
- `经典肉类 + 汤品` 仍使用主食材配对权重
- 存在额外 section 时，会为其独立抽出一道菜
- 新增未知 section 时，也会进入抽样结果
- 冬季限定规则在汤品中继续生效

## 非目标

本次不做：
- 多道素菜之间的真正配对约束启用
- UI 中手动开关分类
- 将 section 配置从 Markdown 完全抽离
